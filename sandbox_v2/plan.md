# Sandbox v2 — Implementation Plan

> **Status:** Draft. Generated by `/plan` from a fresh-rewrite brief with v1
> learnings folded in. See `sandbox/OVERVIEW.md` for the v1 design this
> supersedes.

## Goal

Run a Home Assistant integration's setup, config flow, entities, services,
and events fully inside an **isolated process** ("sandbox"), while the main
HA instance keeps a **single, unified view** of devices, entities, services,
and events that looks identical to running everything locally.

A user adding a light integration through the frontend should end up with:

- A device + entities in the main instance's registries
- The ability to target those entities by area (`light.turn_on` against an
  area resolves them like any other light)
- The integration's entity services + domain-prefixed events available on
  main
- No knowledge that the integration is actually running in a child process

…with the integration code only ever running inside the sandbox.

## Scope

### In scope

- Fresh rewrite under `sandbox_v2/` (Python client) and
  `homeassistant/components/sandbox_v2/` (HA Core integration).
  v1 (`sandbox/` + `homeassistant/components/sandbox/`) stays untouched
  for reference and comparison.
- Routing of new and existing config entries to the correct sandbox based
  on **rules computed at runtime** (no manifest changes, no user config).
- Config flow execution **inside** the sandbox, with form data marshalled
  in/out over the websocket. Main owns the canonical `ConfigEntry` store.
- Entity bridging: devices, entities, capabilities, state, and entity
  services from the sandbox appear in main's registries and service
  registry as if local.
- Event bridging: events fired in the sandbox whose name starts with
  `<owned_domain>_` are forwarded to main's event bus.
- Three sandbox groups out of the box:
  - **`main`** — everything that can't be sandboxed (system integrations,
    integrations using deny-listed platforms, `script` / `automation` /
    `scene`)
  - **`built-in`** — all other built-in integrations
  - **`custom`** — all custom integrations
- Approved-domains gate per sandbox: a sandbox only accepts config entries,
  forwards events, and starts config flows for domains it owns.
- Scoped sandbox auth (token only authorizes `sandbox_v2/*` commands).
- Opt-in core-data sharing (sandbox sees nothing from main by default).
- Store API routing (sandbox `Store` reads/writes go through main).
- Graceful shutdown with restore-state collection from sandbox entities.
- Test infrastructure: pytest plugin variants that exercise the sandbox
  end-to-end and a compat lane for existing core integration tests.

### Out of scope (deferred — track as separate work)

- Service integrations passing **generator payloads** (STT/TTS audio,
  conversation strings). These will be added to a deny-list; integrations
  that use them run in `main`. Sandboxing them is future work.
- YAML-only integrations (no config entry → no sandbox).
- Frontend UI changes. Routing is transparent; surfacing "which integration
  runs where" is a follow-up.
- Migration tooling from v1 sandbox config entries to v2. v1 stays as-is;
  users opt into v2 by enabling it.
- Hot-reload of sandbox processes without HA restart.

## Architecture decisions (resolved)

| Decision | Choice |
|---|---|
| **v2 vs v1** | Fresh rewrite. v1 removed 2026-05-28 (recover from git history). |
| **Assignment rules** | Computed at runtime from manifest + platform inspection. No new user config. |
| **Frontend scope** | Backend-only. Frontend is unchanged; integration setup transparently routes. |
| **v1 limitations** | All addressed in v2 (auth scope, data isolation, Store routing, graceful shutdown). |
| **State sync** | Main subscribes to sandbox's state stream (symmetric with how main → sandbox sharing works). |
| **Service/event mirroring** | Sandbox pushes registrations to main at registration time; dynamic registrations notify explicitly. |
| **Platform compat** | Explicit deny-list (`stt`, `tts`, `conversation`, `assist_satellite`, `wake_word`, `image`, `script`, `automation`, `scene`, …). |
| **Entity bridge** | **Spike both approaches in Phase 1** and decide before Phase 5. See "Open architectural choice" below. |

## Architectural choice — RESOLVED (Phase 1 spike, see [docs/entity-bridge-decision.md](docs/entity-bridge-decision.md))

**Decision: Option B — action-call forwarding.** Proxy entity methods on
main translate to `services.async_call("<domain>", "<service>",
target={"entity_id": [...]})` over a shared `sandbox_v2/call_service`
channel. Sandbox's normal service dispatcher resolves the target and
invokes the entity method.

Headline numbers from the spike (100-entity area `light.turn_on`,
in-process transport, 5 iterations median):

- **Option A:** ~46 ms / call, ~0.46 ms/entity
- **Option B:** ~64 ms / call, ~0.64 ms/entity (~0.18 ms/entity extra)

Per-domain glue is essentially identical (42 vs 48 LOC). The decision
came down to protocol surface: the `sandbox_v2/call_service` channel has
to exist anyway for Phase 6's service mirroring, so Option B reuses it
where Option A adds a parallel entity-only channel.

### Important follow-up the spike surfaced

Some integrations have **non-idempotent service handlers** — the handler
does significant work (e.g., `ai_task._resolve_attachments` fetches bytes
and writes temp files) *before* calling the entity method, and the entity
method's kwargs don't satisfy the original service schema any more.
Neither bridge option handles this cleanly; both are intercepting at the
wrong layer. Resolution paths (out of scope for v2 phases 1–11):

1. **Service-handler-level interception** — forward raw service data to
   the sandbox before the handler runs; sandbox-side handler runs against
   sandbox-local entities. Small extension of the Phase 6 channel.
2. **Sandbox-aware integrations** — integrations cooperate with the
   bridge (e.g., `ai_task` detects sandbox-resident targets and routes
   raw attachments there before resolution).

**Immediate consequence:** `ai_task` and `image` join `ALWAYS_MAIN` in
Phase 2 (see updated rule list). Reason for `image`: its entities expose
bytes-returning methods that downstream integrations like `ai_task` call
locally; sandboxing it would hit the same byte-channel gap that already
deny-lists `camera`.

### Phase 6 — Service & event mirroring ✅ COMPLETE

Approved-domains gate, service-registration mirror, and event mirror
land sandbox→main. A shared `ApprovedDomains` refcounted set is grown
by `EntryRunner` on each successful `entry_setup` and by `EntityBridge`
on each `register_entity`; the `ServiceMirror` and `EventMirror`
consult it before forwarding anything. `ServiceMirror` listens to
`EVENT_SERVICE_REGISTERED` / `EVENT_SERVICE_REMOVED` and pushes
`sandbox_v2/register_service` / `..._unregister_service`; on main,
`SandboxBridge` installs a forwarder that ships each call back over
the shared `sandbox_v2/call_service` channel (with the Phase 5
exception translation reused) but never clobbers an existing handler
— so the `light.turn_on` registered by the host `light`
EntityComponent for our proxy entities stays in charge. `EventMirror`
uses a `MATCH_ALL` listener with an internal-events deny-list to
forward only `<approved_domain>_*` events (e.g. `zha_event`,
`mqtt_message_received`) via `sandbox_v2/fire_event`; main re-fires
each on its own bus so listeners (notably `automation`) react as if
the integration ran locally. **No core HA files touched** — the
Phase 4 `router` hook and the Phase 5 `async_register_remote_platform`
hook are reused unchanged.

### Phase 7 — Scoped sandbox auth & opt-in data sharing ✅ COMPLETE

Adds the `scopes` primitive to `RefreshToken`, enforces it per-command
in the websocket dispatcher, and issues sandbox-scoped access tokens
from a dedicated system user per group. Opt-in core-data sharing lands
as `SandboxGroupConfig` (`share_states` / `share_entity_registry` /
`share_areas`) — default off everywhere, default on for `built-in` and
`main`, default off for `custom`. The runtime accepts the matching
`--share-*` CLI flags into a `SharingConfig` dataclass; consumers
(subscriptions to main's bus) hang off this in later phases. The
sandbox→main websocket connection itself is still future work — the
sandbox runtime is told the URL + scoped token but does not yet open
the connection, so the locked-down posture is enforced trivially today
(no subscription code exists) and the scope-enforcement test exercises
the dispatcher directly via a hand-built scoped token.

### Phase 9 — Graceful shutdown & restore state ✅ COMPLETE

Sandbox runtime handles `sandbox_v2/shutdown` by unloading every loaded
entry via `config_entries.async_unload`, collecting a JSON-safe
snapshot of `RestoreStateData.async_get_stored_states()` (run through
orjson's HA-aware encoder so `Fragment` / `State` / `datetime` survive
the channel), and returning the payload in the reply. The runtime
schedules its `_shutdown` event via `call_soon` *after* the reply is
queued so the subprocess exits 0 on its own. Main's
`SandboxManager.async_graceful_shutdown_all` fans out `MSG_SHUTDOWN` to
every running sandbox, hands each reply to a configurable
`on_shutdown_reply` callback, and waits for the process to exit;
`async_stop_all` then handles SIGTERM/SIGKILL for any sandbox that
didn't ack. The integration's `_on_stop` listener wires both halves
together — Phase 9's `on_shutdown_reply` persists the `restore_state`
payload through `SandboxBridge._handle_store_save` so it lands at the
same `<config>/.storage/sandbox_v2/<group>/core.restore_state` path the
next boot reads from. On startup the runtime warm-loads that file by
swapping `RestoreStateData.store` with a `RemoteStore` instance
(restore_state.py captures `Store` at import time, so Phase 8's
module-attribute rebinding can't reach it) and calling
`async_load()` before any handlers register, so the first
`RestoreEntity.async_get_last_state()` sees the previous run's state.
**No core HA files touched.**

- [x] HA Core shutdown sequence:
  1. Main starts shutdown.
  2. Main sends `sandbox_v2/shutdown` to each sandbox via
     `SandboxManager.async_graceful_shutdown_all`.
  3. Each sandbox: calls `config_entries.async_unload` for each loaded
     entry, collects `RestoreEntity` state, returns the JSON-safe
     payload in the shutdown reply, then sets its shutdown event so
     `run()` exits cleanly.
     *(The `Store.async_save` flush + `EVENT_HOMEASSISTANT_FINAL_WRITE`
     fire from the plan are deferred — they would re-enter
     `channel.call(MSG_STORE_SAVE, …)` inside the shutdown handler
     while the same channel's reader task is dispatching that handler,
     i.e. deadlock. Shipping restore state in the reply is the
     workaround; pending-`delay_save` writes from integrations that
     rely on FINAL_WRITE are not flushed on sandbox shutdown today.
     A concurrent channel dispatcher would lift this — flagged for a
     follow-up.)*
  4. Main writes restore state to its own storage
     (`<config>/.storage/sandbox_v2/<group>/core.restore_state`) via
     `on_shutdown_reply` → `SandboxBridge._handle_store_save`.
  5. Main waits up to the manager's `shutdown_grace` (default 10s),
     then `async_stop_all` falls through to SIGTERM, then SIGKILL.

### Phase 12 — 9b: Concurrent channel dispatcher ✅ COMPLETE

Both `Channel` classes now spawn one task per inbound call/push instead
of awaiting the handler inline, so a handler that re-enters with
`channel.call(...)` no longer pins the reader. A bounded
`asyncio.Semaphore` (default cap 16, overridable via the new
`max_inflight` keyword) gates how many handlers can run at once — the
reader keeps draining the wire while the (cap+1)th handler waits at
the semaphore, so a misbehaving integration can't starve us by fanning
out unbounded inbound work. `close()` cancels every in-flight handler
task and awaits them before returning, so the existing
"cancel mid-flight" behaviour is preserved. With the dispatcher now
concurrent, `_run_graceful_shutdown` fires
`EVENT_HOMEASSISTANT_FINAL_WRITE` and drains the bus so
`async_delay_save`-using `Store`s flush their pending data through
`RemoteStore` (the re-entrant `MSG_STORE_SAVE` each flush issues no
longer deadlocks against the shutdown handler). **No core HA files
touched.**

## Risks

- **Core changes are unavoidable.** Phases 4, 5, and 7 each add a small
  hook into core (`config_entries.py`, `entity_component.py`,
  `auth/models.py`). Each one should be a standalone, reviewed PR before
  v2 lands. Iron Law: do not monkey-patch private internals — v1's
  direct write to `EntityComponent._platforms` is the cautionary tale.
- **Performance under fan-out.** A 200-light area call should not stall
  the loop on websocket round-trips. The Phase 1 spike must measure
  this; if either bridge option exceeds ~50ms for 100 entities, plan a
  batching layer in Phase 5.
- **Service contract drift.** When core changes a service handler (e.g.,
  `light.turn_on` validation), the proxy entity's capability serialization
  must stay in sync. Mitigation: dedicated test that compares "what the
  proxy reports" vs "what the real entity reports" across all 32 domains.
- **Auth scoping is a security boundary.** Phase 7 is the riskiest from
  a security review angle. Get the websocket dispatcher change reviewed
  by someone outside the sandbox work.
- **Deny-list rot.** The `SANDBOX_INCOMPATIBLE_PLATFORMS` set will drift
  as HA adds new platforms. Mitigate: a test that asserts every
  `Platform` enum member is either in the deny-list or covered by a
  proxy entity class.

## Open questions to revisit

1. Should the sandbox token's `scopes` mechanism (Phase 7) be a general
   HA feature or a sandbox-private one? General is cleaner but adds
   surface area. Lean general.
2. Default `share_states` per group: `False` everywhere (locked down) or
   `True` for `built-in` (matches today's behaviour)? Decide before
   Phase 7 lands.
3. ~~Phase 1 spike timing~~ — resolved; spike complete.
4. What's the migration story for users on v1 sandbox today? Today =
   stays on v1; eventually we need a script to flip `__sandbox_group`
   on existing entries.
5. **Non-idempotent service handlers** (surfaced by Phase 1 spike): when
   do we tackle service-handler-level interception or sandbox-aware
   integration hooks? `ai_task` / `image` are punted to `ALWAYS_MAIN`
   for v2; this is a v3 / post-Phase-11 spec.
