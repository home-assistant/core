How It Works

Entity Registration Flow

  1. Integration calls async_add_entities([entity1, entity2, ...])
  2. RemoteClientEntityPlatform intercepts the call — adds entities locally, then registers each with the host via sandbox/register_entity
  3. Host's RemoteHostEntityPlatform creates a proxy entity (e.g., SandboxLightEntity) and adds it to the domain's EntityComponent
  4. Proxy appears in registries, dashboards, automations — indistinguishable from a local entity

State Update Flow

  1. Real entity in sandbox updates its state (e.g., light brightness changes)
  2. RemoteClientEntityPlatform listens for EVENT_STATE_CHANGED and sends sandbox/update_state
  3. Host's proxy entity caches the values and calls async_write_ha_state()
  4. HA Core propagates the state change to the frontend, automations, etc.

Service Call Flow

All service calls in the sandbox are forwarded to the host via SandboxServiceRegistry. The host handles them in one of two ways:

A) Entity method call (e.g., light.turn_on targeting a proxy entity)

  1. User (or automation) calls light.turn_on targeting a proxy entity
  2. HA's service handler invokes async_turn_on() on the proxy
  3. Proxy calls _forward_method("async_turn_on") → sends command via subscribe_entity_commands subscription
  4. RemoteClientEntityPlatform receives the command, dispatches to the real entity's async_turn_on()
  5. Real entity executes (talks to hardware/API), updates state
  6. State update flows back via sandbox/update_state
  7. Sandbox sends entity_command_result → proxy's future resolves

B) Sandbox-registered service (e.g., input_boolean.reload)

  1. Sandbox integration registers a service via SandboxServiceRegistry.async_register()
  2. SandboxServiceRegistry registers locally and sends sandbox/register_service to the host
  3. Host creates a proxy service handler for that domain/service
  4. When the proxy service is called on the host, it sends the call back to the sandbox via subscribe_entity_commands
  5. Sandbox's SandboxServiceRegistry.async_execute_forwarded_call() runs the local handler
  6. Result sent back via sandbox/service_call_result → host's future resolves

Config Entry Grouping

  1. Config entries set options["sandbox"] = "group_name" (a string value)
  2. The sandbox integration discovers all entries with matching group string
  3. Entries sharing the same string run in the same sandbox process
  4. One auth token + one subprocess per group

Constraints

  • Domain isolation: All integrations of the same domain must run in the same sandbox. Services are registered per-domain and the handler needs access to all active data for that domain. You cannot split one domain across host and sandbox.
  • Config-entry only: Only config-entry-based integrations can run in the sandbox. YAML-only integrations are out of scope.
  • Same sandbox grouping: Config entries sharing the same sandbox property value run in one process. One auth token + one subprocess per group.

Known Limitations / Future Work

  • Exception type preservation (mostly fixed, ~10 remaining): ServiceNotSupported, ServiceValidationError with translation metadata, and MultipleInvalid are now correctly preserved through both websocket hops. Remaining issues: domain-specific exception subclasses (e.g. NotValidPresetModeError), TemplateError, and ValueError cannot be reconstructed from websocket error codes.
  • Context forwarding (fixed): Full Context (id, user_id, parent_id) is now forwarded through the sandbox/call_service WS command. Permission checks and context tracking work correctly.
  • Reload support (9 integrations): Reload/reconfig is not supported in sandbox mode. Need a reload protocol where host notifies sandbox to re-fetch and re-setup.
  • Config flows: Config flows do not run in the sandbox. Discovery, setup, and reconfiguration still happen in the host process.
  • Proxy integrations: Integrations that act as proxies (e.g., Bluetooth) need individual support inside the sandbox integration. Each proxy type requires its own bridge implementation.
  • Shutdown / restore state: When HA shuts down, sandboxes should collect restore-state data from entities and push it to the host before exiting.
  • Store persistence: Integrations use Store for persistent data. These should route through the sandbox websocket so the host owns all persistent state.
  • Stream/binary platforms: Camera, STT, TTS, and image platforms return binary data or streams that don't serialize well over websocket.
  • Custom integrations: Future goal. Current focus is built-in integrations.
